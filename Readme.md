# Менеджер конфигурации (Configuration Manager)

## 1. Постановка задачи
В рамках разработки современных приложений часто возникает необходимость централизованного управления глобальными настройками (например, строками подключения к базам данных, параметрами внешнего вида или лимитами подключений). 

**Прикладная проблема:** Если различные компоненты и модули системы будут создавать свои собственные копии менеджера настроек, это неминуемо приведет к рассинхронизации данных. Возникнут ситуации, когда один модуль изменил конфигурацию (например, переключил язык интерфейса), а другой модуль продолжает использовать устаревшие данные. Кроме того, множественное чтение конфигурационных файлов с диска или базы данных ведет к излишнему расходу ресурсов.

## 2. Описание выбранного паттерна
Для решения данной проблемы в проекте реализован порождающий паттерн проектирования **Singleton (Одиночка)**.

**Цель применения паттерна:** Гарантировать, что у класса `ConfigurationManager` существует строго один экземпляр во всем приложении, и предоставить к нему глобальную точку доступа.

**Какую задачу решает Singleton в контексте менеджера конфигурации:**
1. **Единый источник истины (Single Source of Truth):** Все подсистемы приложения обращаются к одному и тому же объекту в памяти. Изменение настройки в любом месте программы мгновенно становится доступным для всех остальных модулей.
2. **Экономия ресурсов:** Инициализация (которая в реальном приложении сопровождается тяжелыми операциями ввода-вывода при чтении JSON/XML файлов) происходит только один раз.
3. **Безопасный доступ:** В данной реализации используется `Lazy<T>`, что дополнительно гарантирует потокобезопасность создания объекта даже в многопоточной и высоконагруженной среде.

## 3. GitHub Flow
Разработка велась в соответствии с GitHub Flow: ветвление для новых фич (`feature/singleton-impl`), атомарные коммиты с понятными описаниями и последующее слияние (Merge) в основную ветку `main`.
